# Canvas app source — `Brichaus Property Ops`

`*.pa.yaml` screen source for the internal property console, authored to the
[Power Apps pa.yaml v3.0 schema](https://raw.githubusercontent.com/microsoft/PowerApps-Tooling/refs/heads/master/schemas/pa-yaml/v3.0/pa.schema.yaml).

```
src/
  App.pa.yaml              StartScreen + brand color globals (OnStart)
  scrProperties.pa.yaml    Screen 1 — list: search, status filter, gallery
  scrPropertyDetail.pa.yaml Screen 2 — read-only scoring + editable ops + Save
  scrAddProperty.pa.yaml   Screen 3 — search-or-request (no bare create)
```

## Important: how these become a running app

Per Microsoft's docs, `.pa.yaml` is **generated by Power Apps Studio and treated
as read-only** — there is **no supported "hand-write YAML → `pac pack` → app"**
path, and `pac canvas pack` is deprecated. These files are (a) the reviewable
**source of truth** in Git and (b) **paste-ready** for Studio. Two supported
ways to instantiate them:

### Option 1 — Code view paste (fastest to see it work)
1. In Power Apps Studio, turn on the **Power Fx formula bar** (App settings), which
   enables **Code view**.
2. Create a blank **tablet** app inside the `BrichausPropertyOps` solution and add
   the Dataverse table **`new_property`** as a data source **aliased `Properties`**
   (rename the data source to `Properties`, or change the alias in the YAML).
3. Add three blank screens named `scrProperties`, `scrPropertyDetail`,
   `scrAddProperty`. Put `App.pa.yaml`'s `OnStart` into **App.OnStart** and set
   **App.StartScreen** to `scrProperties`.
4. For each screen: copy the controls under that screen's `Children:` and **paste
   via Code view** (Ctrl+V) onto the matching screen. Studio **validates and
   normalizes** on paste (it may adjust `Variant`/version and reorder props).

> **Can't I do this in VS Code instead?** No — steps 1–4 (create the app, add the
> data source, make screens, code-view paste) are **Power Apps Studio only**. The
> VS Code Power Platform Tools extension has no canvas designer and no code-view
> paste; `pac` has no "create blank app" command (`pac canvas create` only builds
> from a custom connector, and `pac canvas pack` is deprecated). VS Code's role is
> *after* the app exists: edit/review `.pa.yaml` and run `pac clone`/`pack`/`import`.

### Option 2 — Git integration (for ongoing round-trip)
Connect the solution to source control and let Studio own the `.pa.yaml`. Note
Dataverse Git integration is **Azure DevOps only** (see `docs/04`), so for this
GitHub repo, Option 1 + committing the Studio-exported `.pa.yaml` back here is
the practical loop.

## Confirm-in-Studio before it runs clean

These are the spots most likely to need a one-line fix once IntelliSense sees
your real metadata — all are flagged inline in the YAML too:

- [ ] **Choice labels (no global name needed).** The screens are global-name-free
      — they use `Choices(<column>)`, `drp.Selected`, and label text
      (`.Value = "Managed"`). The only assumption is that the option **labels** are
      `Managed / Prospect / Pending / Lost` and `100% / 50% / 20%`. If a label
      differs, change the literal string. If `.Value` ever returns blank on your
      choices, use the numeric-value fallback in `docs/05-studio-paste-guide.md`.
- [ ] **Data source alias** is `Properties` (= `new_property`).
- [ ] **Toggle control.** Uses classic `Toggle` (`.Value`). If Studio gives you a
      modern Toggle, change `.Value` → `.Checked` in the Save `Patch`.
- [ ] **Column security.** The scoring/fee fields are read-only by design and
      (per `docs/02`) column-secured — the app only *displays* them; don't bind
      them to inputs.

## Deliberate deviations from `docs/03` (and why)

- **Explicit input controls + explicit `Patch`** instead of an Edit Form with
  auto-generated data cards. Hand-authored data cards are fragile; explicit
  controls paste cleanly and make the field-by-field `Patch` (and the status↔bit
  sync) auditable.
- **Property type shown read-only** from `cr55d_propertytype_rentcast` text — no
  `new_propertytypeclassification` lookup, keeping this a single-table app.
- **Long detail screen** is laid out with absolute Y positions; in Studio you may
  wrap the ops section in a scrollable container if it overflows your canvas.

## What's intentionally NOT here

Units, property contacts, and agreement terms (the related/link tables) — out of
scope for this single-table build (see `docs/01` scope note).
